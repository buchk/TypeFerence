# 0005 — Cross-implementation conformance suite design

**Status:** Accepted (2026-07-08)

## Context

With two implementations, "the spec is the source of truth" is only enforceable if
something machine-checks that both implementations produce identical bytes from
identical source. Neither implementation's own tests can prove that; a shared,
language-neutral fixture corpus can.

## Decision

- Fixtures live at the repository root under `conformance/fixtures/<NNN-name>/`,
  each with a `source/` tree and a `manifest.json` describing build options and the
  expected outcome. Both implementations run the full corpus in CI
  (`go test ./conformance` and `ConformanceSuiteTests`); the `conformance` CI job
  fails if either implementation disagrees with a recorded digest.
- **The contract is artifacts, not CLI prose.** Success fixtures record the
  `typeference-directory-v1` digest of each emitted top-level target directory.
  Error fixtures record only that compilation must fail with a diagnostic;
  identical error strings across implementations are a non-goal (shared,
  stable diagnostic codes are future work).
- **Digests are generated by one implementation and verified by the other.**
  `go test ./conformance -update` regenerates manifests; the C# runner then has to
  independently reproduce every digest for the suite to be green. A hand-edited
  digest cannot pass both sides unless it is actually correct.
- `conformance/.gitattributes` disables git text conversion under `fixtures/`:
  several fixtures pin CRLF, missing-trailing-newline, and BOM behavior at the
  byte level and would be silently destroyed by autocrlf.
- Corpus at introduction (25 fixtures): minimal and maximal composition, five-level
  embedding chains, shallowest-wins promotion, same-depth ambiguity (resolved and
  error), capability promotion/override/ambiguity, interface embedding and
  structural satisfaction, unicode and escape-table content, CRLF/no-trailing-
  newline/BOM byte edges, number-token preservation, promoted-skill context
  isolation, ASCII slot-name enforcement, duplicate ids, cycles, profile-embeds-
  agent, and trust manifests (unsigned, fail-closed required-signature, externally
  signed with the signature map outside the source root).

## Consequences

- Any change to canonical output requires regenerating manifests in the same
  commit as the spec change and ADR that justify it; unexplained digest churn is
  visible in review.
- New spec rulings must land with a fixture (ADR-0004 rulings all have one).

## Alternatives considered

- **Byte-comparing stderr for error fixtures** — rejected: freezes diagnostic
  wording exactly when we want implementations free to improve messages.
- **Committing full expected output trees for every fixture** — rejected as the
  primary mechanism: digests are compact and the committed `dist/` tree already
  provides one fully materialized reference example that both implementations
  byte-compare against in their own test suites.
- **A standalone runner binary** — rejected: each implementation's native test
  framework is already wired into CI and reports failures with context.
